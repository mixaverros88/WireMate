Stub Mapping Properties

A comprehensive reference of every property available when defining a WireMock stub mapping. Covers top-level fields, request matchers, response definitions, delays, faults, proxying, transformers, scenarios, metadata, and webhooks.

Overview

#overview

A stub mapping is a JSON object POSTed to /__admin/mappings. It pairs a request pattern (what to match) with a response definition (what to return). Additional top-level fields control priority, persistence, scenarios, metadata, and post-serve actions like webhooks.

💡
Quick Reference
Only request and response are required. Everything else is optional and lets you fine-tune matching, add delays, simulate faults, or drive stateful scenarios.

Full Example

#full-example

A comprehensive stub mapping demonstrating many of the available properties.

Complete Stub Mapping 
{
  "id": "76ada7b0-a177-4f21-a3ab-c03c78a2e004",
  "name": "Create user endpoint",
  "priority": 1,
  "persistent": true,

  "scenarioName": "user-flow",
  "requiredScenarioState": "Started",
  "newScenarioState": "user-created",

  "request": {
    "method": "POST",
    "urlPath": "/api/users",
    "headers": {
      "Content-Type": { "equalTo": "application/json" },
      "Authorization": { "matches": "Bearer .*" }
    },
    "queryParameters": {
      "verbose": { "equalTo": "true" }
    },
    "bodyPatterns": [
      { "matchesJsonPath": "$.name" },
      { "equalToJson": "{\"role\":\"admin\"}", "ignoreExtraElements": true }
    ]
  },

  "response": {
    "status": 201,
    "statusMessage": "Created",
    "headers": {
      "Content-Type": "application/json",
      "X-Request-Id": "{{randomValue type='UUID'}}"
    },
    "jsonBody": { "id": 1, "name": "Alice" },
    "fixedDelayMilliseconds": 150,
    "transformers": ["response-template"]
  },

  "metadata": {
    "team": "backend",
    "sprint": 12
  }
}

Top-Level Properties

#top-level

These properties sit at the root of the stub mapping JSON object. Only request and response are required to create a functional stub.

id string (UUID) Optional Server-assigned or user-provided unique identifier ▾

A UUID that uniquely identifies this stub mapping. If omitted when creating a stub, WireMock will generate one automatically. You can supply your own UUID if you want deterministic IDs for version control or idempotent updates.

Example
"id": "76ada7b0-a177-4f21-a3ab-c03c78a2e004"
name string Optional Human-readable label for the stub ▾

A descriptive name for the stub. Has no functional effect on matching but is extremely useful for identifying stubs in the admin UI, logs, and API responses.

Example
"name": "Get all users - happy path"
priority integer Optional Lower number = higher priority (default: 5) ▾

When multiple stubs match an incoming request, the one with the lowest priority number wins. The default priority is 5. Use priority 1 for specific overrides and higher numbers for catch-all fallbacks.

Example
"priority": 1   // highest priority — matches before default (5)
persistent boolean Optional Survive resets (default: false) ▾

When true, this stub survives calls to POST /__admin/mappings/reset and POST /__admin/reset. Non-persistent stubs are wiped on reset. Stubs loaded from JSON files on disk are implicitly persistent.

Example
"persistent": true
request object Required Request matching criteria ▾

Defines the criteria an incoming request must satisfy for this stub to match. Contains URL matching, HTTP method, headers, query parameters, cookies, and body patterns. See the Request Matching sections below for all sub-properties.

response object Required Response to return when matched ▾

Defines the HTTP response WireMock returns when this stub matches. Includes status code, headers, body content, delays, faults, and proxy behavior. See the Response Definition sections below for all sub-properties.

Scenario Properties

#scenarios-props

Scenarios enable stateful behaviour. A stub can be conditional on the current state of a named scenario and can transition it to a new state when matched.

scenarioName string Optional Assigns the stub to a named scenario ▾

The name of the scenario this stub belongs to. All stubs sharing the same scenario name participate in the same state machine. Scenarios always start in the Started state.

Example
"scenarioName": "order-flow"
requiredScenarioState string Optional Only match when scenario is in this state ▾

The scenario must be in this state for the stub to match. If omitted, the stub matches regardless of scenario state. The initial state is always Started.

Example
"requiredScenarioState": "Started"
newScenarioState string Optional Transition scenario to this state after match ▾

After this stub matches, the scenario transitions to this new state. Subsequent requests will then be matched against stubs requiring the new state. If omitted, the scenario state remains unchanged.

Example
"newScenarioState": "user-created"

Metadata

#metadata-props

Arbitrary key-value data you can attach to a stub for tagging, filtering, and bulk operations.

metadata object Optional Arbitrary tags for find/remove operations ▾

A freeform JSON object. You can store any key-value pairs here. Metadata is searchable via POST /__admin/mappings/find-by-metadata and POST /__admin/mappings/remove-by-metadata. Common uses include tagging stubs with team names, test run IDs, or environment labels.

Example
"metadata": {
  "team": "backend",
  "testRunId": "run-2024-001",
  "environment": "staging"
}

Request — URL & Method

#request-basic

Every request matcher needs at least a URL strategy. Use exactly one of the four URL properties. The method field restricts which HTTP verb to match.

method string Optional HTTP method to match (default: ANY) ▾

The HTTP method the incoming request must use. If omitted, any method matches. Use ANY explicitly to match all methods.

Accepted Values
GET POST PUT DELETE PATCH HEAD OPTIONS TRACE ANY
url string Use one Exact full URL match (path + query string) ▾

Matches the entire URL including path and query string. The match is exact and case-sensitive. Use this when you need to match a specific path with specific query parameters.

Example
"url": "/api/users?status=active&page=1"
urlPath string Use one Exact path match (ignores query string) ▾

Matches only the path component, ignoring any query parameters. This is the most commonly used URL matcher. Combine with queryParameters for selective query matching.

Example
"urlPath": "/api/users"
urlPattern string (regex) Use one Regex match on full URL (path + query string) ▾

Matches the full URL (path + query) against a Java-style regular expression. The regex must match the entire URL, not just a substring.

Example
"urlPattern": "/api/users/[0-9]+\\?.*"
urlPathPattern string (regex) Use one Regex match on path only (ignores query string) ▾

Matches only the path component against a regex, ignoring query parameters. The most flexible URL matcher for dynamic path segments.

Example
"urlPathPattern": "/api/users/[0-9]+"
ℹ️
URL Matching Rules
Use exactly one URL property per request. If you need path matching with query parameter constraints, use urlPath or urlPathPattern together with the queryParameters object.

Request — Headers & Cookies

#request-headers

Match on specific HTTP headers or cookies. Each entry uses a string matcher operator (see Matchers section).

headers object Optional Header matchers keyed by header name ▾

An object where each key is a header name and the value is a matcher operator. All specified headers must match for the stub to trigger. Header names are case-insensitive.

Example
"headers": {
  "Content-Type": { "equalTo": "application/json" },
  "Authorization": { "matches": "Bearer .*" },
  "X-Debug": { "absent": true }
}
cookies object Optional Cookie matchers keyed by cookie name ▾

An object where each key is a cookie name and the value is a matcher operator. Works identically to header matching but targets cookies from the Cookie header.

Example
"cookies": {
  "session": { "matches": "[a-f0-9]{32}" },
  "debug": { "contains": "verbose" }
}

Request — Query Parameters

#request-query

Match specific query string parameters using the same matcher operators as headers.

queryParameters object Optional Query param matchers keyed by param name ▾

An object where each key is a query parameter name and the value is a matcher. Best used with urlPath or urlPathPattern (not url which already includes the query string).

Example
"queryParameters": {
  "page": { "equalTo": "1" },
  "search": { "contains": "alice" },
  "sort": { "absent": true }
}

Request — Body Patterns

#request-body

An array of matchers applied to the request body. All patterns must match (AND logic). Supports string matching, JSON matching, XML matching, JSONPath, XPath, and binary comparison.

bodyPatterns array Optional Array of body matcher objects ▾

Each element is a matcher object. All elements must match for the stub to trigger. The available body matcher operators are listed below.

OperatorTypeDescription
equalTo string Exact string match on the full body
contains string Body must contain this substring
matches string (regex) Body matches this regex pattern
doesNotMatch string (regex) Body must NOT match this regex
equalToJson string (JSON) Semantic JSON equality (order-insensitive for objects)
matchesJsonPath string (JSONPath) Body satisfies a JSONPath expression
equalToXml string (XML) Semantic XML equality (ignores whitespace & attribute order)
matchesXPath string (XPath) Body satisfies an XPath expression
binaryEqualTo string (base64) Exact binary match (base64-encoded expected value)
absent boolean Body must be empty / absent
JSON Body Matching
"bodyPatterns": [
  {
    "equalToJson": "{\"name\":\"Alice\",\"role\":\"admin\"}",
    "ignoreArrayOrder": true,
    "ignoreExtraElements": true
  }
]
JSONPath Matching
"bodyPatterns": [
  { "matchesJsonPath": "$.name" },
  { "matchesJsonPath": "$[?(@.age > 18)]" },
  {
    "matchesJsonPath": {
      "expression": "$.name",
      "equalTo": "Alice"
    }
  }
]
XML / XPath Matching
"bodyPatterns": [
  {
    "matchesXPath": "/users/user[@id='1']",
    "xPathNamespaces": {
      "ns": "http://example.com/ns"
    }
  }
]
💡
equalToJson Modifiers
When using equalToJson, add "ignoreArrayOrder": true to ignore array element order and "ignoreExtraElements": true to allow the request to contain additional fields not in the expected JSON.

Request — Basic Auth

#request-auth

A shorthand for matching HTTP Basic Authentication credentials without manually decoding the Authorization header.

basicAuthCredentials object Optional Match username/password from Basic auth ▾

Matches the decoded username and password from the Authorization: Basic ... header.

PropertyTypeDescription
username string Expected username
password string Expected password
Example
"basicAuthCredentials": {
  "username": "admin",
  "password": "secret123"
}

Request — Multipart

#request-multipart

Match individual parts of a multipart/form-data request.

multipartPatterns array Optional Matchers for multipart form data parts ▾

An array of part matchers. Each matcher can constrain the part name, headers, and body using the same matcher operators available elsewhere.

PropertyTypeDescription
matchingType string ANY (at least one part matches) or ALL (every part matches)
headers object Header matchers for the part (e.g., Content-Disposition)
bodyPatterns array Body matchers applied to the part body
Example
"multipartPatterns": [
  {
    "matchingType": "ANY",
    "headers": {
      "Content-Disposition": {
        "contains": "name=\"file\""
      }
    },
    "bodyPatterns": [
      { "binaryEqualTo": "iVBORw0KGgo..." }
    ]
  }
]

Request — Custom Matcher

#request-custom

Delegate request matching to a custom Java extension registered with WireMock.

customMatcher object Optional Invoke a custom request matcher extension ▾

Invokes a custom matcher extension by name, passing optional parameters. The extension must be registered in WireMock's classpath.

PropertyTypeDescription
name string The registered name of the custom matcher extension
parameters object Arbitrary parameters passed to the matcher
Example
"customMatcher": {
  "name": "my-ip-matcher",
  "parameters": {
    "allowedCidrs": ["10.0.0.0/8", "192.168.0.0/16"]
  }
}

String Matcher Operators

#matchers

These operators are used wherever a value needs to be matched — headers, cookies, query parameters, and body patterns. Each matcher object uses exactly one operator.

OperatorTypeDescription
equalTo string Exact string equality. Add "caseInsensitive": true for case-insensitive match.
matches string (regex) Value matches the Java regex pattern
doesNotMatch string (regex) Value does NOT match the regex (negative match)
contains string Value contains the given substring
absent boolean The header/param/cookie must NOT be present (set to true)
equalToJson string (JSON) Semantic JSON equality. Supports ignoreArrayOrder and ignoreExtraElements.
matchesJsonPath string / object JSONPath expression that must evaluate to a non-empty result. As an object, supports sub-matcher via expression + operator.
equalToXml string (XML) Semantic XML equality (ignores whitespace, attribute order). Supports enablePlaceholders.
matchesXPath string / object XPath expression. Supports xPathNamespaces for namespace-aware matching.
binaryEqualTo string (base64) Exact binary comparison. Value is base64-encoded.
before string (datetime) Date/time value is before the given ISO-8601 timestamp
after string (datetime) Date/time value is after the given ISO-8601 timestamp
equalToDateTime string (datetime) Date/time value equals the given ISO-8601 timestamp
ℹ️
Modifier Properties
Matchers can include modifier flags alongside the operator: "caseInsensitive": true (for equalTo, contains), "ignoreArrayOrder": true (for equalToJson), "ignoreExtraElements": true (for equalToJson), "enablePlaceholders": true (for equalToXml), "xPathNamespaces" (for matchesXPath), "truncateExpected" / "truncateActual" (for datetime matchers), "expectedOffset" / "expectedOffsetUnit" (for relative datetime).

Response — Status & Body

#response-basic

The core response properties: HTTP status code and the body content. Use exactly one body property per response.

status integer Required HTTP status code (e.g., 200, 404, 500) ▾

The HTTP status code to return. Any valid HTTP status code is accepted (100-599).

Example
"status": 200
statusMessage string Optional Custom HTTP reason phrase ▾

Overrides the default HTTP reason phrase (e.g., "OK", "Not Found"). Most modern HTTP clients ignore this, but it can be useful for legacy systems.

Example
"statusMessage": "Everything is awesome"
body string Use one body type Plain text or escaped string body ▾

The response body as a plain string. Use this for text, HTML, or escaped JSON. For JSON responses, prefer jsonBody.

Example
"body": "Hello, World!"
jsonBody object / array Use one body type JSON body as a native object (no escaping) ▾

The response body as a native JSON value. No string escaping needed — the object is serialized to JSON automatically. This is the preferred way to return JSON.

Example
"jsonBody": {
  "id": 1,
  "name": "Alice",
  "roles": ["admin", "user"]
}
base64Body string (base64) Use one body type Binary body as base64-encoded string ▾

A base64-encoded binary body. Use for returning images, PDFs, or other binary content directly from the stub definition.

Example
"base64Body": "iVBORw0KGgoAAAANSUhEU..."
bodyFileName string Use one body type Serve body from a file in __files/ ▾

Path to a file in WireMock's __files/ directory. The file contents become the response body. Ideal for large responses or binary files. Path is relative to the __files root.

Example
"bodyFileName": "responses/users-list.json"

Response — Headers

#response-headers

HTTP headers to include in the response.

headers object Optional Response headers as key-value pairs ▾

An object where keys are header names and values are header values (strings). For multiple values, use an array of strings. Supports response template syntax when the response-template transformer is active.

Example
"headers": {
  "Content-Type": "application/json",
  "Cache-Control": "no-cache",
  "X-Request-Id": "{{randomValue type='UUID'}}",
  "Set-Cookie": ["a=1", "b=2"]
}

Response — Delays

#response-delay

Simulate network latency by adding delays before the response is sent. Use either a fixed delay or a distribution for more realistic behavior.

fixedDelayMilliseconds integer Optional Fixed delay in ms before responding ▾

A fixed delay in milliseconds applied before the response is sent. Cannot be combined with delayDistribution.

Example
"fixedDelayMilliseconds": 500
delayDistribution object Optional Randomized delay from a distribution ▾

A randomized delay sampled from a statistical distribution. More realistic than a fixed delay. Three distribution types are available.

Uniform Distribution

Random delay evenly distributed between lower and upper bounds.

PropertyTypeDescription
type"uniform"Distribution type identifier
lowerintegerMinimum delay in milliseconds
upperintegerMaximum delay in milliseconds
Uniform
"delayDistribution": {
  "type": "uniform",
  "lower": 50,
  "upper": 300
}
Log-Normal Distribution

Models real-world latency: most responses are fast, some are slower. Uses median and sigma.

PropertyTypeDescription
type"lognormal"Distribution type identifier
medianintegerMedian delay in milliseconds
sigmanumberStandard deviation (higher = more variance)
Log-Normal
"delayDistribution": {
  "type": "lognormal",
  "median": 100,
  "sigma": 0.4
}
Fixed Distribution

An explicit fixed delay expressed as a distribution object.

PropertyTypeDescription
type"fixed"Distribution type identifier
millisecondsintegerFixed delay in milliseconds
Fixed
"delayDistribution": {
  "type": "fixed",
  "milliseconds": 200
}
chunkedDribbleDelay object Optional Trickle response bytes over time ▾

Sends the response body in chunks over a specified duration, simulating a slow connection. The total number of chunks and the total time can be configured.

PropertyTypeDescription
numberOfChunksintegerNumber of chunks to split the body into
totalDurationintegerTotal time in ms to send all chunks
Example
"chunkedDribbleDelay": {
  "numberOfChunks": 5,
  "totalDuration": 1000
}

Response — Fault Simulation

#response-fault

Simulate network-level failures instead of returning a normal HTTP response. When fault is set, no body/status/headers are sent.

fault string Optional Simulate a network fault ▾

Instead of returning a normal response, WireMock will simulate a network-level failure. This is invaluable for testing client resilience and error handling.

Accepted Values
EMPTY_RESPONSE MALFORMED_RESPONSE_CHUNK RANDOM_DATA_THEN_CLOSE CONNECTION_RESET_BY_PEER
ValueBehavior
EMPTY_RESPONSEReturns a completely empty response (no status line, no headers, no body), then closes the connection
MALFORMED_RESPONSE_CHUNKSends an OK status header, then garbage data, then closes the connection
RANDOM_DATA_THEN_CLOSESends random bytes, then closes the connection without a valid HTTP response
CONNECTION_RESET_BY_PEERImmediately resets the TCP connection (RST)
Example
"response": {
  "fault": "CONNECTION_RESET_BY_PEER"
}

Response — Proxying

#response-proxy

Instead of returning a canned response, proxy the request to a real backend service.

proxyBaseUrl string (URL) Optional Forward request to this base URL ▾

When set, WireMock forwards the matched request to this base URL and returns the real response. The request path is appended to the proxy base URL. This turns the stub into a selective proxy.

Example
"response": {
  "proxyBaseUrl": "https://api.real-service.com"
}
additionalProxyRequestHeaders object Optional Extra headers to add when proxying ▾

Headers to add or override on the proxied request before sending it upstream. Only used when proxyBaseUrl is set.

Example
"additionalProxyRequestHeaders": {
  "X-Forwarded-By": "WireMock",
  "Authorization": "Bearer real-token"
}
removeProxyRequestHeaders array Optional Headers to strip before proxying ▾

A list of header names to remove from the request before forwarding to the proxy target.

Example
"removeProxyRequestHeaders": ["Cookie", "X-Internal-Token"]

Response — Transformers

#response-transform

Apply server-side transformations to the response before sending it to the client. The most common transformer is response-template, which supports Handlebars expressions.

transformers array of strings Optional List of transformer extension names to apply ▾

An array of registered transformer names. Transformers are applied in order. The built-in response-template transformer enables Handlebars-based dynamic responses using request data, random values, dates, etc.

Example
"transformers": ["response-template"]
transformerParameters object Optional Parameters passed to transformers ▾

A freeform object whose contents are passed to the active transformers. Custom transformer extensions can read these parameters to modify their behavior.

Example
"transformerParameters": {
  "greeting": "Hello",
  "includeTimestamp": true
}
💡
Response Templating
With "transformers": ["response-template"], you can use Handlebars expressions in body and headers: {{request.path}}, {{request.headers.Authorization}}, {{randomValue type='UUID'}}, {{now}}, {{jsonPath request.body '$.name'}}, and more.

Webhooks / Post-Serve Actions

#webhooks

Trigger additional HTTP requests (webhooks or callbacks) after a stub has been matched and the response has been sent. Requires the webhook extension.

postServeActions object Optional Actions to run after response is sent ▾

An object keyed by action name (e.g., webhook). Each value configures the action. The webhook action makes an HTTP request to a specified URL after the stub response is delivered.

PropertyTypeDescription
urlstringTarget URL for the webhook callback
methodstringHTTP method (GET, POST, PUT, etc.)
headersobjectHeaders to send with the webhook request
bodystringBody to send (supports templating)
delayobjectDelay before firing: {"type":"fixed","milliseconds":1000}
Example
"postServeActions": {
  "webhook": {
    "url": "https://hooks.example.com/notify",
    "method": "POST",
    "headers": {
      "Content-Type": "application/json"
    },
    "body": "{\"event\":\"stub-matched\",\"path\":\"{{request.path}}\"}",
    "delay": {
      "type": "fixed",
      "milliseconds": 500
    }
  }
}
